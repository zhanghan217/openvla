# OpenVLA 挖掘机适配方案讨论

## 1. 机械臂兼容性

**问**: 我的机械臂是6个点的，是lerobot那种，可以用这个模型吗，需要改的地方多不多？

**答**: 可以用，但需要 fine-tuning。
- 模型默认输出 7-DoF（6-DoF位姿 + 夹爪），取前6个就行
- 关键是要在自己的机械臂数据上 LoRA 微调（~100条演示数据即可），否则泛化效果差
- 动作归一化参数（`unnorm_key`）需要改成机械臂的工作空间范围
- 改动量不大，主要就是用 `vla-scripts/finetune.py` 跑微调，数据格式转成 RLDS 即可

---

## 2. 挖掘机场景微调方案

**问**: 数据是在真实挖掘机上采的，挖掘机只有三个关节（大臂、小臂、铲斗），数据是ROS格式。

**方案**:

### 步骤1: 数据处理 ROS → RLDS
- 从 rosbag 提取图片和关节角度，保存为每帧一个样本
- 每个样本包含: image + 3维action (大臂, 小臂, 铲斗)
- 用 tfrecord 格式，参考 https://github.com/kpertsch/rlds_dataset_builder

### 步骤2: 注册数据集
- `prismatic/vla/datasets/rlds/oxe/configs.py` — 添加数据集配置（action_dim=3）
- `prismatic/vla/datasets/rlds/oxe/transforms.py` — 添加数据转换函数
- `prismatic/vla/datasets/rlds/oxe/mixtures.py` — 添加混合配置

### 步骤3: LoRA 微调
```bash
torchrun --standalone --nnodes 1 --nproc-per-node 1 vla-scripts/finetune.py \
  --vla_path "openvla/openvla-7b" \
  --data_root_dir <数据目录> \
  --dataset_name <数据集名> \
  --run_root_dir <输出目录> \
  --adapter_tmp_dir <临时目录> \
  --lora_rank 32 \
  --batch_size 4 \
  --grad_accumulation_steps 4 \
  --learning_rate 5e-4 \
  --image_aug False
```

### 步骤4: 推理
- `unnorm_key` 改成挖掘机关节角度的归一化范围（min/max）
- 输出取前3维直接作为关节指令发布到 ROS

**关键点**: 约100条演示数据就够了，控制频率建议5-10Hz，避免采集过程中的停顿动作。

---

## 3. 语言指令是否需要

**问**: 微调的时候，是不是需要把指令也给模型？

**答**: 对。OpenVLA 是 **图片 + 语言指令 → 动作** 的模型。

训练时每个样本包含：
- **image**: 当前摄像头画面
- **instruction**: 语言指令，比如 `"把大臂抬起来"` / `"收小臂"` / `"铲斗向前推"`
- **action**: 对应的关节角度

所以采集数据时，每条轨迹需要标注对应的语言指令。如果没标，就得回去给每条轨迹补上指令文本。

---

## 4. BridgeData V2 数据集介绍

**问**: BridgeData V2 数据集是什么样子的？

| 项目 | 内容 |
|---|---|
| **规模** | 60,096 条轨迹，约 400GB |
| **机器人** | WidowX 250 6DOF 机械臂 + Robotiq 2F-85 夹爪 |
| **场景** | 24 个环境（玩具厨房、水槽、桌面等） |
| **技能** | 13 类：抓放、推、叠、折叠、开门/抽屉、扫等 |
| **控制频率** | 5Hz |
| **平均轨迹长度** | 38 步 |

**每条轨迹包含：**
- **RGB 图像**: 640x480，主要视角是"肩上固定相机"
- **深度图**: 来自固定相机（部分数据）
- **动作**: 7-DoF 末端执行器增量（x, y, z 位移 + 旋转 + 夹爪开合）
- **语言指令**: 自然语言标注，如 `"把胡萝卜放到盘子上"`
- **本体感知**: 关节状态

**关键特点**: 所有轨迹都用自然语言标注（不是固定任务 ID），模型可以接受开放式文本指令。

---

## 5. 挖掘机数据格式

**问**: 第一是否需要将节点的动作与视频相对应？第二如果训练完了给一个更复杂的任务，模型能理解需要将两个动作组合吗？

### 问题一: 动作和视频要对应吗？
不需要特意区分"哪个关节在动"。OpenVLA 是逐帧的：
```
每一步: 图片(当前画面) + 指令("挖沙子") → 动作[大臂, 小臂, 铲斗]
```
只需要保证：**每一帧的图片和那一时刻三个关节的角度对齐就行**。

### 问题二: 能组合复杂任务吗？
不能。OpenVLA 是**反应式**模型，一条指令 → 一个动作序列，不会自动组合。
- 训练了"挖沙子"和"装车"，模型学会两个独立技能
- 但"把沙子挖进车里"，模型**不会自己拆分成先挖再装**
- **解决办法**: 直接采集"把沙子挖进车里"的完整轨迹数据，标注对应指令，一起训练

---

## 6. 帧级动作格式

**问**: 给OpenVLA训练的数据集，关节动作也是帧级的吗？

**答**: 对，就是这样。每条数据是一个序列：
```
第1帧: 图片(挖掘机某个姿态) + 指令("挖沙子") → 动作[大臂+1°, 小臂-2°, 铲斗+0°]
第2帧: 图片(下一个姿态)   + 指令("挖沙子") → 动作[大臂+1°, 小臂-1°, 铲斗+3°]
第3帧: 图片(...)          + 指令("挖沙子") → 动作[大臂+0°, 小臂+0°, 铲斗+5°]
...
```

动作是**增量**（这一帧到下一帧关节变化了多少），不是绝对角度。模型学到的就是：看到这个画面 + 这个指令 → 该输出什么增量动作。

---

## 7. openvla-7b 预训练模型能力

**问**: openvla-7b 是预训练模型吗，能实现什么功能？

**答**: 是预训练模型。

**直接能用的：**
- 在 **WidowX 机械臂 + BridgeData V2 场景**上零样本执行任务（给图片+语言指令，输出7-DoF动作）
- 比如 "把胡萝卜放到盘子上"、"翻转锅" 等桌面操作任务

**不能直接用的：**
- 挖掘机场景（完全不同的机器人、关节结构、工作环境）
- 任何 WidowX 训练数据之外的机械臂

**预训练的意义：**
模型已经学会了"看图 + 听指令 → 输出动作"这个通用能力。挖掘机微调是在这个基础上**适配**，而不是从零训练，所以只需要 ~100 条数据就能学会，比从头训练快得多、数据需求少得多。

简单说：**预训练给你通用能力，微调给你专用能力。**

## 8. OpenVLA 工作原理

**问**: 给它装一个摄像头，它捕捉当前姿态（一张图片），然后分析？还是根据实时的视频流输入，处理每一帧来做动作？

**答**: **单帧，不是视频流。**

工作流程：

```
摄像头拍一帧图片
    ↓
连同语言指令一起输入模型
    ↓
模型输出一个动作（关节增量）
    ↓
机械臂执行这个动作
    ↓
再拍一帧 → 再推理 → 再执行 → 循环
```

是**闭环控制**，每一步都是：
1. 看一眼当前画面（1张图）
2. 决定下一步做什么
3. 执行
4. 回到第1步

控制频率建议 5-10Hz，即每秒执行 5~10 次这个循环。不是把整个视频丢给模型分析，而是每一步只看"现在长什么样"。

---

## 9. 模型如何判断动作完成

**问**: 比如挖沙子动作，那它怎么判断动作完成了？也是当时训练的时候知道的吗？

**答**: 模型**不知道**什么时候完成。它只做一件事：看了当前画面 → 输出下一步动作。它没有"任务完成"的概念。

实际运行中是这样的：
```
第1帧: 画面是铲斗在空中 → 输出: 大臂下压，铲斗下翻
第2帧: 画面是铲斗刚接触沙面 → 输出: 铲斗继续翻，小臂收
第3帧: 画面是铲斗已经插入沙子 → 输出: 铲斗上翻，大臂抬起
...继续推理，直到你手动停止
```

**模型会一直输出动作，停不停是你的程序决定的。**

常见做法：
- **固定步数**：训练时挖沙子平均30步，推理时也跑30步就停
- **人工干预**：操作员看着，觉得到位了就发停止指令
- **加个判断器**：额外写一个简单的检测程序，比如铲斗角度到了某个值就停，或者视觉检测沙子已经装进铲斗了就停

所以需要在模型之外再加一个**结束条件**的逻辑。

## 10. 模型如何区分哪个关节在动

**问**: 挖掘机的ROS数据包比如只包括大小臂和铲斗的倾角，模型要怎么知道是哪个臂在动？因为大臂动，小臂和铲斗也一定会产生角度变化。

**答**: 模型**不需要**知道"哪个臂在动"。

它输出的是**所有关节的增量同时输出**：
```
输入: 当前图片 + "挖沙子"
输出: [大臂-1°, 小臂+2°, 铲斗+3°]  ← 三个值同时给出，不管谁动谁不动
```

"大臂动，小臂和铲斗也会有角度变化"——没关系。模型看到的是**整体状态**，输出的是**整体动作**。

示例：
```
第1帧: 大臂高、小臂伸、铲斗空 → 动作: [-2, +1, +3]  (大臂下压为主)
第2帧: 大臂降了、小臂微动、铲斗略翻 → 动作: [-1, +0, +5]  (铲斗翻为主)
第3帧: 大臂到位、小臂收、铲斗插入沙 → 动作: [+1, -2, +2]  (大臂抬+小臂收)
```

模型不需要做"这是大臂在动还是小臂在动"这种判断。它只是学到了：**在这个画面下、要做这个任务，三个关节各应该动多少。** 角度变化是三个关节一起的，输出也是三个关节一起的，一一对应就行。

ROS 数据里只要每帧同时记录了三个关节的倾角，就能直接用。

---

## 11. 是否需要底层控制参数

**问**: 控制挖掘机肯定是驾驶室的操作台，或者接线直接控制各个臂。是否能实现准确控制各个臂的倾角？训练时是否需要把关节动作再加上液压压力等参数？

**答**: **不需要加液压压力这些底层参数。**

OpenVLA 工作在**关节空间**，不关心底层怎么驱动：
```
模型输出: [大臂-1°, 小臂+2°, 铲斗+3°]
    ↓
你的控制程序负责翻译成:
    大臂液压缸 → 给多少压力/电流
    小臂液压缸 → 给多少压力/电流
    铲斗液压缸 → 给多少压力/电流
```

这层翻译是你自己写的，和模型无关。模型只管"该动多少度"，不管"怎么让液压缸动"。

**训练时只需要：**
- 图片
- 语言指令
- 三个关节的角度（倾角）

**不需要：**
- 液压压力
- 电流
- 油温
- 速度
- 任何底层控制参数

这样做的好处是：换一台挖掘机（液压系统不同），模型照样能用，因为它只输出角度，底层驱动是你自己适配的。

---

## 12. 角度输出到实际控制的链路

**问**: 它只输出角度，但是也得让控制系统知道需要做什么才能输出这些角度啊。

**答**: 对，但那是你写的控制程序的事，不是模型的事。

整个系统是这样的：
```
OpenVLA 模型
    ↓ 输出: [大臂-1°, 小臂+2°, 铲斗+3°]
你写的控制程序（你自己负责）
    ↓ 翻译成具体指令
液压系统执行
    ↓
挖掘机动作
    ↓
摄像头拍新一帧
    ↓
回到 OpenVLA
```

你写的控制程序需要做的事情：
- 接收模型输出的3个角度增量
- 转换成液压系统的控制信号（电流、PWM、CAN指令等，取决于硬件）
- 发给执行器

这个控制程序你得自己写，OpenVLA 不管这部分。模型只负责"决策"（该动多少度），不负责"执行"（怎么让液压缸动起来）。就像人告诉挖掘机司机"大臂往下2度"，司机自己知道怎么操作手柄。

---

## 13. 控制程序的翻译角色

**问**: 我们要准备一个控制程序，比如大臂+1度，控制某个模块给电压+1等等，然后这个程序收到模型输出的动作，去进行操作？

**答**: 对，理解完全正确。就是两层：

```
第一层: OpenVLA → "大臂+1度"（决策层）
第二层: 你的控制程序 → "给大臂液压阀加X伏电压"（执行层）
```

你的控制程序就是中间的翻译器，把模型的"意图"变成硬件能理解的"电信号"。

---

## 14. 系统延迟问题

**问**: 这样延迟不是很高吗？

**答**: 延迟确实有，但对挖掘机来说够用。

各环节耗时估算：
| 环节 | 耗时 |
|---|---|
| 摄像头拍照 | ~10ms |
| 图片预处理 | ~5ms |
| **OpenVLA 推理（7B模型）** | **~80-150ms** |
| 控制信号发送 | ~5ms |
| 液压系统响应 | ~50-100ms |
| **总计** | **~150-270ms** |

对应控制频率大约 **4-7Hz**，和训练时的 5Hz 匹配。

**为什么对挖掘机够用：**
- 挖掘机是慢速设备，一秒钟移动几度很正常
- 不需要毫秒级响应（不像平衡机器人、无人机）
- 5Hz 完全能满足挖掘、装车等操作

**如果嫌慢：**
- 用更小的模型（OpenVLA 也有更小版本）
- 用 INT8/INT4 量化加速推理
- 用 TensorRT 部署优化

但一般挖掘场景下，当前方案的延迟不是问题。

## 15. 端到端与执行层的关系

**问**: 决策规划和执行岂不是又分开了，就不是彻底的端到端自动了？

**答**: 没有分开。端到端的意思是：

```
像素 → 动作（关节角度）
```

OpenVLA 已经做到了这一层，它就是端到端的。

"控制程序"（把角度转成电压）相当于**神经到肌肉**的关系：
```
人脑决定"抬手"（端到端决策）
    ↓
神经信号传到肌肉（自动的，你不用管）
    ↓
手臂抬起
```

没有人会觉得"人控制手臂不是端到端的，因为中间过了神经"。神经就是执行层，永远存在。

**任何机器人系统都有这一层**，包括最"端到端"的系统：

| 系统 | 决策层 | 执行层 |
|---|---|---|
| OpenVLA | 模型输出角度 | 你的液压控制程序 |
| Tesla Bot | 模型输出角度 | 电机驱动器 |
| ALOHA | 模型输出角度 | 电机驱动器 |

区别只在于：**你的控制程序是手写的规则，还是也用学习的方法。**

对于挖掘机，手写就够了——角度到电压是简单的映射关系，不需要学习。真正需要学习的"看图决策"部分，OpenVLA 已经做了。

---

## 16. 挖掘机现有控制器的利用

**问**: 每个关节挖掘机是有个控制器的。

**答**: 对，挖掘机每个关节本身就有控制器（液压阀/电控系统）。控制程序其实很简单：

```
OpenVLA 输出: [大臂+1°, 小臂-2°, 铲斗+3°]
    ↓
直接发给各个关节控制器:
    大臂控制器 → 目标角度+1°
    小臂控制器 → 目标角度-2°
    铲斗控制器 → 目标角度+3°
    ↓
控制器自己闭环执行到位
```

甚至不需要自己写电压转换，直接给每个关节控制器发目标角度就行，控制器自己搞定液压驱动。整个链路更简洁。

## 17. 伏羲挖掘机数据集分析与微调方案

### 数据集概况

数据集路径: `/mnt/nas/创新中心/伏羲数据集/`

包含两个子集：
- **excavator-video**: 1000 条完整装车视频（10辆卡车，每辆100条），仅视频无关节数据
- **excavator-motion**: 92214 条轨迹，含图片+关节角度+动作，可直接用于 OpenVLA 微调

### excavator-motion 数据结构

| 字段 | 维度 | 说明 |
|---|---|---|
| `/observations/images/main` | [N, 480, 640, 3] | BGR 主视角图像 |
| `/observations/images/elevation` | [N, 200, 200, 3] | 激光雷达高程图 |
| `/observations/qpos` | [N, 4] | 当前关节角度 (大臂, 小臂, 铲斗, 回转) |
| `/action` | [N, 4] | 下一帧关节角度 |
| `/observations/qvel` | [N, 4] | 关节速度 |
| 帧率 | 10fps | 完美匹配 5-10Hz 要求 |

当前本地数据: 3台挖掘机 (75, 306, 490)，各10个 H5 文件，总计 30 个文件、75,385 帧、74GB。已完整下载。

### 微调方案

#### 步骤1: 数据预处理
将每个 H5 文件转换成每帧一条样本：
```
每帧: 图片(RGB, 224×224) + 指令("挖掘装车") + 动作增量[Δ大臂, Δ小臂, Δ铲斗, Δ回转]
```

关键处理：
- BGR → RGB 转换
- 图片缩放到 224×224
- 动作增量 = action[t] - qpos[t]（下一帧角度 - 当前角度）
- 给每条轨迹加语言指令，如 `"挖掘装车"`

#### 步骤2: 转 RLDS 格式
用 [rlds_dataset_builder](https://github.com/kpertsch/rlds_dataset_builder) 打包成 TFRecord/RLDS 格式。

#### 步骤3: 注册数据集
- `prismatic/vla/datasets/rlds/oxe/configs.py` — 添加 action_dim=4
- `prismatic/vla/datasets/rlds/oxe/transforms.py` — 添加归一化函数
- `prismatic/vla/datasets/rlds/oxe/mixtures.py` — 添加挖掘机混合配置

#### 步骤4: LoRA 微调
```bash
torchrun --standalone --nnodes 1 --nproc-per-node 1 vla-scripts/finetune.py \
  --vla_path "/home/wika/data/openvla/openvla-7b" \
  --data_root_dir <RLDS数据目录> \
  --dataset_name excavator_motion \
  --run_root_dir <输出目录> \
  --adapter_tmp_dir <临时目录> \
  --lora_rank 32 \
  --batch_size 4 \
  --grad_accumulation_steps 8 \
  --learning_rate 5e-4 \
  --image_aug False
```

#### 步骤5: 推理
```python
inputs = processor("In: What action should the robot take to 挖掘装车?\nOut:", image)
```

## 18. OpenVLA 数据格式与模型输出详解（代码分析）

### 数据格式要求

训练时每步需要的字段（`datasets.py:38-67`）：

| 字段 | 说明 | 代码出处 |
|---|---|---|
| `observation.image_primary` | RGB 图像 | `datasets.py:42` |
| `task.language_instruction` | 语言指令文本 | `datasets.py:42` |
| `action` | 动作向量（增量） | `datasets.py:40` |
| `dataset_name` | 数据集名称 | `datasets.py:40` |

- **深度图**: `load_depth=False`（`datasets.py:96`），**不需要**
- **本体感知**: `load_proprio=False`（`datasets.py:97`），**不需要**
- **窗口**: `window_size=1`（`datasets.py:103`），每样本是单帧，不是多帧窗口
- **未来动作**: `future_action_window_size=0`，没有动作分块

### 模型输出

**单步输出，不是30步**（`modeling_prismatic.py:506-536`）：

```
每次推理: 图片 + 指令 → 1个动作增量向量
```

- `max_new_tokens = action_dim`（比如4维关节就生成4个token）
- 动作是**相对增量**，不是绝对角度
- 默认7-DoF末端执行器格式 `[dx, dy, dz, droll, dpitch, dyaw, gripper]`
- 可配置为任意维度

### 动作离散化

- 256 个均匀 bin 覆盖 [-1, 1]（`action_tokenizer.py:13`）
- 每个动作维度独立离散化
- 离散 token 映射到 LLM 词表最后 256 个 token
- 推理时反向映射：token → bin center → 连续值

### 动作归一化

- 训练: 用 q01/q99 分位数映射到 [-1, 1]，夹爪不归一化
- 推理: 用 `unnorm_key`（数据集名）查找统计量做反归一化
- 归一化统计存在 `dataset_statistics.json`

### 指令格式

OpenVLA 使用 PurePromptBuilder（`base_prompter.py:28-73`）：
```
"In: What action should the robot take to {instruction.lower()}?\nOut: "
```

- 指令自动转小写
- 以 `\nOut:` 结尾
- 模型从冒号后开始生成动作 token

### 与你之前理解的差异

| 之前的理解 | 实际 |
|---|---|
| 输出 30 步动作 | **只输出 1 步**（每帧一个动作增量） |
| 需要深度图 | **不需要** |
| 需要本体感知 | **不需要** |

### 对挖掘机的影响

你的数据只需要：
1. **RGB 主视角图像**（已有，BGR → RGB 即可）
2. **语言指令**（需补充，如"挖掘装车"）
3. **4-DoF 动作增量**（可从 qpos 计算：`action[t+1] - qpos[t]`）
4. **数据集名**（如"excavator_motion"）

高程图、qvel、car_pos 等都用不上，不需要放进数据集。

## 19. 部署推理时的取帧策略

**问**: 模型部署到挖掘机上，是捕捉摄像头画面取帧推理吗？比如30fps视频，想一秒推理10个动作，就三帧取一次？

**答**: 对。核心原则：**推理频率必须匹配训练频率**。

训练数据是 10fps，一个动作增量代表 100ms 内的关节变化。如果用 30fps（33ms 间隔），模型输出的增量会对不上实际需要的变化量。

```
30fps 视频流: 帧1 帧2 帧3 帧4 帧5 帧6 帧7 帧8 帧9 ...
              ↓         ↓         ↓         ↓
推理帧(10Hz): 帧1       帧4       帧7       帧10
              ↓         ↓         ↓         ↓
输出动作:    动作1     动作2     动作3     动作4
```

每次: 推理 → 执行 → 等下一帧 → 再推理，循环。每个动作间隔 100ms。

---

## 20. Hz 解释

**Hz（赫兹）= 每秒执行次数**

- **10Hz** = 每秒 10 次 = 每 100ms 一次
- **30fps** = 每秒 30 帧 = 每 33ms 一帧

推理频率要和训练频率一致，因为模型学到的是"100ms 内该动多少"。如果推理改成 50ms 间隔（20Hz），模型输出的增量还是按 100ms 算的，实际只过了 50ms，动作就会偏大——机械臂会冲过头。
### 数据集优势
- 10fps 帧率正好匹配 OpenVLA 推荐频率
- 4-DoF 动作空间简洁清晰（大臂、小臂、铲斗、回转）
- 已有多视角图像（主视角+高程图）
- 已有 LeRobot 转换工具可参考
- 75,385 帧数据量充足，远超 100 条轨迹的最低要求

## 17. VLA 中的 LLM 是负责预测动作，还是只做编码传给 Action 模块？

在 OpenVLA 中，当你说"把杯子拿起来"，LLM（Llama 2 7B）是**直接预测动作**的，而不是只做编码。

具体流程：

1. **视觉编码**：图像经过 DINOv2 + SigLIP 双编码器，再通过 MLP 投影器映射到语言模型的嵌入空间。
2. **指令处理**：语言指令"把杯子拿起来"被 tokenize 后作为普通文本 token 输入 LLM。
3. **动作预测（核心）**：LLM 的 transformer 层同时融合视觉和语言信息后，**以 next-token prediction 的方式逐 token 生成动作序列**。每个动作维度（7维末端执行器控制）被离散化为 256 个 bin，映射为 LLM 词表中的 token。

所以 LLM 本身既负责"理解"指令含义，又负责"输出"离散化的动作 token。没有一个独立的 Action 模块——LLM 的输出层（token 词表）本身就充当了动作预测头，生成的 token 经逆离散化后得到连续的动作值，直接控制机械臂。

## 18. 既然 LLM 能预测动作，为什么还要训练特定任务？

这正是 VLA 的核心设计思想：**LLM 预训练提供了通用能力，但还不够。**

原因是：LLM 预训练（互联网数据）只给了语言理解和视觉常识，它从未见过"如何移动机械臂"的数据。所以必须在大规模机器人演示数据上进行微调，让 LLM 学会将语言理解**映射到物理动作输出**——这是从语言 token 到动作 token 的领域迁移。

具体分三个层次：

1. **大模型预训练**（OpenVLA 在 OpenX 数据集上微调）：教 LLM "看到这个场景 + 这个指令，大概应该输出什么动作模式"。这一步学到的是跨任务、跨机器人的泛化策略。

2. **少样本微调**（适配新环境）：预训练模型在你特定机器人的具体场景下见过一些演示后，就能快速学会精确执行。比如给它 50 个"拿杯子"的演示，它就能在你的环境里稳定完成。

3. **为什么不能直接零样本用**：预训练时学的是各种场景的混合动作分布，对于你特定的杯子位置、特定的机械臂关节限位、特定的空间布局，它不知道"256 个 bin 里哪个 bin 对应你这个具体动作"。微调就是做这种领域对齐。

类比：就像 GPT-4 已经很聪明，但你还是要对它做 RLHF 才能让它好好聊天；LLM 已经懂语言了，但你还要在机器人数据上训练它才能控制物理世界。

## 19. 动作指令也是 LLM 生成的？

是的，动作 token 是 LLM **直接从它的输出层逐个生成**的，和生成文字 token 完全一样的机制。

整个过程：

```
输入：[图像tokens] + "把杯子拿起来"
                    ↓
              LLM transformer层（7B参数）
                    ↓
输出：a0 a1 a2 a3 a4 a5 a6  （7个离散动作token）
                    ↓
              逆离散化
                    ↓
         [dx, dy, dz, rx, ry, rz, gripper]
                    ↓
              机械臂执行
```

关键点：

- LLM 的**同一个输出头**既能生成文字 token，也能生成动作 token，它们共享同一个词表空间
- 训练时，词表中 256 个最不常用的 token 被**覆盖替换**为动作 token（0-255 对应 7 个动作维度的离散 bin）
- 训练损失只计算在动作 token 上，文字部分只是条件输入，不计算 loss
- 推理时，它就像聊天一样逐 token 采样，只不过输出的不是文字而是动作序列

这就是为什么叫 "Vision-Language-Action"——动作和语言用的是**同一个模型、同一个词表、同一个生成机制**，只是用途不同。

## 20. DM0.5 模型里的 LLM 也是一样的作用吗？

不一样。DM0.5 和 OpenVLA 在架构上有一个**关键区别**：

- **OpenVLA**：LLM 同时负责理解和动作预测，用同一个词表输出动作 token（离散化）。
- **DM0.5**：LLM（Gemma3 4B）**不直接输出动作**，它只负责多模态理解和推理。动作由一个独立的 **680M Action Expert** 用 Flow Matching 生成连续动作。

| | OpenVLA | DM0.5 |
|---|---|---|
| LLM 作用 | 理解 + 动作预测（一体） | 纯理解和推理 |
| 动作生成 | LLM 输出头直接生成离散动作 token | 独立 Action Expert（680M）用扩散模型生成连续动作 |
| 动作空间 | 离散化（256 bins） | 连续动作（Flow Matching） |
| 额外能力 | 无 | 历史记忆（最长60s）+ 具身推理 CoT |

DM0.5 还多了一个 **Embodiment CoT** 机制——LLM 会先输出语言形式的推理过程（任务规划、环境预测、动作意图），这些推理文本作为条件输入给 Action Expert，再由 Action Expert 生成精确的连续动作。

所以 DM0.5 的分工更像：**LLM 当"大脑"思考，Action Expert 当"小脑"执行**。
