## 3. 环境 — tensorflow 替换为 tensorflow-cpu

**问题**: tensorflow (GPU 版) 与 PyTorch 争夺 CUDA context，导致：
- 每步耗时从 12s 逐渐涨到 26s
- 重复报 `Unable to register cuDNN/cuFFT/cuBLAS factory` 错误

**修法**:
```bash
pip uninstall tensorflow tensorflow-datasets tensorflow-metadata tensorflow-graphics -y
pip install tensorflow-cpu==2.15.0
pip install tensorflow-datasets==4.9.3 --no-deps
pip install tensorflow_graphics==2021.12.3 --no-deps
pip install tensorflow-metadata==1.13.1
pip install protobuf==4.25.9
```

**验证**: `TF GPUs: []` — TF 彻底不碰 GPU

## 4. prismatic/vla/datasets/excavator_dataset.py — 性能瓶颈识别

**问题**: `_load_frame()` 每帧调用 `skip(idx).take(1)`，对 11GB TFRecord 做顺序扫描，越往后越慢

**原因**: `tf.data.TFRecordDataset.skip()` 是 O(n) 顺序跳过的，第 10000 步时每次取一帧要先扫描 10000 条记录

**待修**: 初始化时把所有帧缓存在系统内存（~11GB），后续 O(1) 随机访问

## 7. prismatic/vla/datasets/excavator_dataset.py — 全量预加载修复（解决第 4 条遗留问题）

**问题**: 沿用第 4 条分析的结论，`_load_frame()` 每次 `__getitem__` 都重新打开 TFRecord 并 `skip(idx).take(1)` 顺序扫描，随 `idx` 增大耗时线性上升，是 `training_bs4.log` / `training_bs4_v2.log` 中每步耗时从 3s 涨到 26s+ 的根因。另外 `__init__` 里声明了 `self._frames_cache` / `self._cache_size` 但从未使用，是死代码。

**修法**:
- `__init__` 中一次性遍历 TFRecord，把全部帧的 `image_bytes` 和 `action` 解析后存入 `self._records` 列表（不在此时解码成 PIL Image，减少内存占用）
- 删除 `_load_frame()` 方法，`__getitem__` 改为直接按下标 `self._records[idx]` 取值后再 `Image.frombytes` 解码，访问复杂度从 O(idx) 降为 O(1)
- 删除未使用的 `_frames_cache` / `_cache_size` 死代码，删除未使用的 `import struct`

**修改位置**: `prismatic/vla/datasets/excavator_dataset.py`（`__init__`、`__getitem__`，删除 `_load_frame`）

**验证**: 用独立脚本加载真实数据集 `FuXiData/processed/excavator_motion.tfrecord`（75355 帧）
- 全量加载耗时 14.29s，峰值内存 11.25GB（机器可用内存约50GB，足够）
- 在 idx=0/10000/30000/50000/75354 处做随机访问，5 次总耗时 0.002ms，验证 O(1) 访问，不再随 idx 增大而变慢
- `python3 -m py_compile` 语法检查通过
